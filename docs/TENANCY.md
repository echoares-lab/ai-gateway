# Tenancy & Workspace Domain Model

> **Status:** Approved. Foundational domain model for [Roadmap Epic #30 — Multi-tenant workspace management](https://github.com/echoares-lab/ai-gateway/issues/30).
> This document defines the tenant/workspace/team/repo hierarchy, lifecycle states, naming conventions, and mapping rules to ensure secure, traceable isolation across the AI Gateway.

---

## 1. Domain Hierarchy & Terminology

To support multiple organizations, repositories, applications, and developers without naming conflicts or visibility leaks, the AI Gateway adopts a structured, hierarchical tenancy model:

```text
Organization (Tenant)
  └── Workspace
        ├── Team
        │     └── API Keys (Credential pool)
        └── Repository / Application / Client
```

### 1.1 Organizational Units

| Term | Definition | Multiplicity | Primary Identifier |
|---|---|---|---|
| **Organization (Tenant)** | The top-level corporate or business unit (e.g. echoares). Represents the ultimate billing, administrative, and compliance boundary. | 1 | Org Slug (e.g. echoares) |
| **Workspace** | A logical partition within an organization representing a business unit, department, or strategic initiative (e.g., core-ai, fintech). | 1..* | Workspace Slug (e.g. core-ai) |
| **Team** | A group of users/developers working within a Workspace who share access to a specific pool of credentials and resources. | 1..* | Team Slug (e.g. engineering-t1) |
| **Repository / Client** | An individual codebase (e.g., ai-gateway), service, or AI application that consumes the gateway. | 1..* | Git Remote URL / Repo Name |

## 2. Naming Conventions & Lifecycles

Standardized naming ensures consistency across Prometheus metrics, database schemas, and external telemetry (Langfuse).

### 2.1 Workspace and Team Slugs
- Must be lowercase, alphanumeric, and hyphenated ([a-z0-9-]+).
- **Team naming template:** {org}-{workspace}-{team} (e.g. echoares-core-engineering).
- **Workspace naming template:** {org}-{workspace} (e.g. echoares-core-ai).

### 2.2 API Key Labels
Gateway-issued client credentials (passed via Authorization: Bearer <key>) must include lineage metadata encoded in their database descriptions:
- **Format:** ak-{org}-{workspace}-{team}-{repo}-{environment}
- **Examples:**
  - ak-echoares-core-eng-gateway-dev
  - ak-echoares-fintech-sec-audit-prod

### 2.3 Lifecycle States

```text
[ Proposed ] --> [ Active ] --> [ Cooldown / Deprecated ] --> [ Suspended ]
```

1. **Proposed:** Defined in configuration/spec but not yet materialized (no database tables or active keys).
2. **Active:** Fully provisioned with active database roles, LiteLLM keys, and allowed workspace rules.
3. **Cooldown/Deprecated:** Active keys exist but are restricted to read-only or read-only/no-write, rejecting new prompt generation request kinds.
4. **Suspended:** Keys are temporarily disabled or revoked in the database. Usage statistics are preserved for audit logs.

## 3. Artifact & Telemetry Mapping

Every runtime artifact and trace generated by the gateway must map directly back to a specific node in the tenancy hierarchy.

### 3.1 Telemetry & Tracing (Langfuse)
All downstream LLM calls proxied by the translator or processed by LiteLLM must attach metadata tags to Langfuse traces:
- tenant_id: The Organization/Tenant slug.
- workspace_id: The Workspace slug.
- team_id: The Team slug.
- repo_name: The client repository name.
- environment: dev, staging, or prod.

### 3.2 LiteLLM Configuration Mapping
LiteLLM Virtual Keys and Teams represent the primary runtime enforcement mechanisms:
- **LiteLLM Team:** Maps directly to a Team ({org}-{workspace}-{team}). Used to track collective rate limits (RPM, TPM) and dollar budgets.
- **LiteLLM Key:** Maps directly to a Repository / Client in a specific environment.

### 3.3 Repo bootstrap (`setup-repo-env.sh`)
Per-repo CLI isolation:

```bash
setup-repo-env.sh --org echoares --workspace core --team eng --env dev /path/to/repo
```

The script writes `CODEX_HOME`, gateway URLs in `.envrc`, and documents the expected LiteLLM key label `ak-{org}-{workspace}-{team}-{repo}-{environment}`. Use `gen-client-config.sh` with the same tenant flags for client snippets.

## 4. Visibility & Isolation Boundaries

Multi-tenancy enforces isolation across four major dimensions:

1. **Routing Isolation:** Orgs/Workspaces can define custom fallback preferences (e.g., forcing Workspace A to fall back only to Gemini, while Workspace B falls back only to Claude).
2. **Data & Cache Isolation:** Cache keys are isolated by the client's Authorization token hash. Cache entries cannot be read across workspace boundaries.
3. **MCP Tool Access:** Workspaces can restrict client access to specific MCP servers or read-only postgres schemas based on the Team's metadata payload.
4. **Quota and Budget Boundaries:** Budgets are defined at the Team level and enforced natively by LiteLLM's token/dollar budget limiters.

## 5. Explicit Non-Goals & Child Issues

To maintain context efficiency and velocity, the following are explicitly marked as **non-goals** for this initial modeling stage:
- Writing SQL/Prisma migrations to alter the SQLite or Postgres database schemas.
- Adding runtime tenancy check middleware to the FastAPI translator.
- Deploying the frontend admin dashboard or user roles.
